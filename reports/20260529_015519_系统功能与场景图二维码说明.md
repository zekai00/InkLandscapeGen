## 2. 诗词解析与场景图扩充的真实实现

当前系统演示版不是调用训练好的场景图扩充模型，也不是 LLM 真正理解诗词后生成 JSON，而是规则流程：

```text
诗词输入
  -> 关键词匹配，抽取显式意象
  -> 根据山水画常识补全隐含意象
  -> 生成实体、属性、关系、布局四类结构化信息
  -> 将结构化摘要拼入图像生成提示
```

示例输入：

```text
明月松间照，清泉石上流。
```

示例输出：

```text
显式实体：山石、水体、树木、明月
隐含补全：薄雾
属性：远近层次、虚实留白、传统笔墨、清冷夜景
关系：水体沿山石流动、明月照映松林
布局：远景布置远山与云雾、中景组织水面或溪流、前景以树木和山石形成遮挡
```

因此，论文或答辩中更准确的说法是“系统实现了轻量级规则解析与场景图补全演示”，不宜说成完整复现第四章或第五章算法。

## 3. 二维码问题

之前不能生成真实二维码的原因是后端 `app/qrcode.py` 依赖第三方包 `qrcode`，但当前 Python 环境没有安装该包。路由捕获异常后返回了占位图，因此前端看到的不是可扫码二维码。

已完成修复：

- 安装 `qrcode[pil]`。
- 在 `system/backend/requirements.txt` 中记录依赖。
- 生成接口返回真实二维码 base64。
- 前端生成信息区域展示图片二维码。

已验证：

```text
qrcode_base64_len=708
```

## 4. 可继续增强的功能

短期最值得加的功能：

- 生成历史记录：保存输入、风格、参数、seed、耗时、图片路径和场景图。
- 场景图编辑：用户可在生成前修改实体、属性、关系和布局，再提交生成。
- 任务队列：生成任务较慢时返回 task_id，前端轮询进度，避免请求长时间阻塞。
- 异常与显存提示：捕获 OOM、模型未加载、端口不可达等错误，并给前端明确提示。
- 图像管理：提供删除、重新生成、按 seed 复现、下载图片和下载二维码。
- 对比展示：同一输入下展示不同 seed、风格或参数的多图对比。

两天内更现实的增强顺序：

```text
1. 历史记录与图片管理
2. 场景图编辑
3. 任务状态轮询与异常提示
4. 多图对比展示
```
