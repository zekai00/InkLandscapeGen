# 系统工作台现代化与工程增强报告

更新时间：2026-05-29 02:39:20

## 一、调整目标

本次调整围绕答辩意见中“系统工程深度不足、界面展示较弱、缺少并发与异常处理说明”等问题展开。目标不是单纯换皮，而是把系统从原先的单页调用界面改为面向论文展示的完整工作台：既能完成诗词输入、语义解析、场景结构编辑、图像生成，也能展示任务队列、历史管理、多图对比和显存/异常提示。

## 二、前端界面改造

前端主文件已重建为：

- `/root/Workspace/ControllableImageGeneration/system/frontend/src/App.vue`

新的前端采用 Vue 3 + Element Plus，不额外引入新的前端依赖。界面结构由单页表单改为四个工作区：

1. **创作台**
   - 输入诗词或画面描述；
   - 选择山水画风格：默认、水墨、青绿、浅绛；
   - 选择构图模式：自动、横幅、竖幅、方幅、远景、近景；
   - 调整尺寸、步数、随机种子、细节强度、留白强度和反向约束；
   - 显示任务进度、阶段状态和生成结果。

2. **历史记录**
   - 从后端 `/api/v1/history` 读取生成记录；
   - 支持复用历史参数；
   - 支持删除历史图像；
   - 支持加入多图对比。

3. **多图对比**
   - 支持最多 4 张图像横向对比；
   - 展示风格、提示词、尺寸、步数和随机种子；
   - 用于答辩中说明不同参数、构图和风格控制对结果的影响。

4. **系统状态**
   - 展示后端服务状态；
   - 展示文本解析服务是否启用；
   - 展示推理设备与显存信息；
   - 展示任务队列中的近期任务。

视觉上去掉了原先较简单的表单式布局，改为纸张质感背景、印章标识、墨线动态效果、半透明工作面板和现代化进度条。首页不再是说明页，而是直接进入可操作的创作工作台，并在未生成新图时展示近期生成图像。

## 三、场景图预览的处理方式

原先“扩充场景图预览”如果只作为 JSON 展示，确实容易显得和论文主线脱节。本次没有删除它，而是改为“论文链路 / 可编辑场景结构”，对应论文第三章、第四章和第五章：

- **第三章 数据层**：展示显式意象、触发词和画面属性；
- **第四章 语义扩充**：展示隐含意象补全、空间关系和布局线索；
- **第五章 生成控制**：将场景结构转化为风格、构图和扩散生成约束。

用户可以直接编辑显式意象、补全意象、画面属性、空间关系、布局安排和风格提示。生成时，前端会把编辑后的结构作为 `scene_graph_override` 提交给后端。这样该模块既是论文算法链路的可视化，也是真实参与生成控制的输入，不再只是展示性组件。

## 四、后端工程增强

后端已经具备以下接口：

- `GET /api/v1/health`：健康检查、风格列表、生成服务状态、显存状态、文本解析状态、队列数量；
- `POST /api/v1/scene-graph`：诗词语义解析与场景结构生成；
- `POST /api/v1/tasks`：创建异步生成任务；
- `GET /api/v1/tasks/{task_id}`：轮询单个任务状态；
- `GET /api/v1/tasks`：新增，返回近期任务列表；
- `GET /api/v1/history`：生成历史记录；
- `DELETE /api/v1/history/{filename}`：删除历史图像；
- `GET /images/{filename}`：读取生成图像。

新增的任务队列采用单任务顺序执行，避免多个生成任务同时抢占显存。前端通过轮询任务状态展示“等待调度、解析场景图、图像生成、结果保存、异常处理”等阶段。后端会对显存不足、连接失败等错误进行分类，向前端返回更适合用户理解的提示。

生成图像保存目录为：

- `/root/static/generated`

这样可以避开项目目录容量限制，便于长期保存演示结果。系统界面和构建产物中没有出现具体模型名称，避免把系统实现绑定到某一个底层模型。

## 七、对论文第六章的建议写法

第六章应从“工程实现”升级为“算法链路落地系统”，重点强调：

1. 系统不是简单 Vue + FastAPI 页面，而是围绕第三章数据、第四章语义结构、第五章生成控制搭建的端到端验证平台；
2. 系统引入任务队列，解决图像生成耗时长、显存占用高、前端阻塞的问题；
3. 系统引入可编辑场景结构，使算法中间结果能够被用户检查和修正；
4. 系统提供历史记录、多图对比和参数复用，使实验展示和创作迭代更完整；
5. 系统提供显存与异常提示，提高工程可用性和答辩时的技术深度。
